我要给开发流程加一个新要求：开发日志系统。请按以下步骤执行：
第一步：创建 devlog 文件夹和说明文件
在项目根目录创建 devlog/ 文件夹，并在里面创建 README.md，内容说明：

这个文件夹是项目的开发日志
每个 Phase 对应一个文件，命名规则 phase-{编号}-{简短主题}.md
日志用于未来整理成技术博客和项目复盘

第二步：创建日志模板
在 devlog/ 下创建 TEMPLATE.md，作为每次写日志的模板。模板应包含以下章节（用 Markdown 标题）：

Phase 基本信息（编号、主题、开始/结束时间、实际耗时）
目标回顾（从 PLAN.md 复制本 Phase 的目标）
实际完成情况（对照 PLAN 的交付物清单,打勾或说明差异）
关键决策记录（做了哪些选择、有哪些候选方案、最终为什么这么选）
遇到的问题与解决过程（报错/设计难点/与 SPEC 冲突，每个问题写：现象 → 分析 → 解决）
关键代码片段（挑 1-3 段最有代表性的代码，带注释说明）
可作为博客素材的亮点（适合写进技术博客的故事或洞察）
复盘与反思（如果重来会怎么做得更好）
给下一个 Phase 的提醒(本 Phase 留下的技术债、未解决的问题、需要注意的点)

同时标注：
日志要真实,包括你犯过的错、绕过的弯路、一开始写错又改掉的代码。不要把过程美化成"一步到位",如果是第二次才写对的,就要写出来第一次写错的原因；
写日志的步骤只能写 devlog/ 下的文件,不得修改任何其他代码或配置文件。如果在回顾中发现代码有问题,把问题记录在日志的"给下一个 Phase 的提醒"里,但不要自己去改；
每份日志 500-1500 字为宜。重要的 Phase 可以写长,简单的 Phase 短一点也行,但关键决策和遇到的问题必须详细。

第三步：更新 PLAN.md
在 PLAN.md 的"通用规则"章节补充一条:每个 Phase 结束、在我确认"通过"之后,你要立刻基于 devlog/TEMPLATE.md 创建本 Phase 的日志文件并填写完成,文件名为 phase-{编号}-{简短主题}.md。日志写完后再等我说"开始下一个 Phase"。
第四步:追溯补写 Phase 1 的日志
Phase 1 已经完成,但还没有日志。请基于你对 Phase 1 的完整记忆,按模板补写 devlog/phase-1-scaffolding.md。尽量真实地写出你当时的思考和遇到的问题,不要编造。如果某些细节已经记不清,诚实标注"此处记忆不清"。

完成这四步后告诉我,然后我们继续。
